Project Overview

Cloud Resource Allocation Using Reinforcement Learning

This project studies how cloud scaling decisions can be recommended from historical workload behaviour. It combines reinforcement learning concepts with a practical web stack so users can submit current system signals and receive a scaling recommendation.

Problem Statement

Cloud systems need enough compute resources to handle demand, but over-provisioning wastes money and under-provisioning harms performance. The goal of this project is to recommend whether the system should scale up, hold, or scale down based on operational inputs such as CPU, memory, instance count, and request rate.

The project focuses on decision support: it helps estimate the next scaling action from workload patterns already learned from historical samples.

Policy Used

The learning idea behind the project is tabular Q-learning, a reinforcement learning policy that evaluates actions by estimating long-term reward for each state-action pair.

During experimentation, the policy learns from workload samples and reward feedback. In the currently deployed system, live requests are answered using the pre-trained sample dataset generated from that RL process, so no new online training is performed when a user submits input.

Development Stack

  • Frontend: React with Vite
  • Gateway Backend: Node.js with Express
  • Inference Backend: Python with FastAPI
  • Data Source: Preprocessed RL sample dataset in JSON form

Theoretical Flow

  1. Historical workload samples are converted into RL-friendly states.
  2. A reward design encourages balanced scaling decisions while penalizing overload and unnecessary cost.
  3. Q-learning is used offline to derive useful scaling behaviour from the dataset.
  4. The resulting learned samples are stored and reused for live recommendation.
  5. At runtime, the system estimates the most suitable action for the user’s current input.

Calibrations Used

  • CPU input calibration: user CPU percentage is scaled against the maximum normalized CPU value found in the trained dataset.
  • Memory input calibration: user memory percentage is scaled against the maximum normalized memory value found in the trained dataset.
  • Request-rate calibration: request rate is scaled proportionally against the maximum normalized request-rate value from the dataset.
  • Instance calibration: instance counts are mapped from the UI range 1 to 3 into the normalized interval 0.0 to 1.0.
  • State matching calibration: live inputs are not retrained online; they are compared against stored RL samples and matched to the closest learned state.

Core Formulae

Q-learning update:

Q(s, a) = Q(s, a) + alpha * [r + gamma * max Q(s', a') - Q(s, a)]

CPU normalization:

cpu_util = (cpu / 100) * cpu_max

Memory normalization:

mem_util = (memory / 100) * mem_max

Instance normalization:

instances_norm = (instances - 1) / 2

Request-rate normalization:

request_rate_norm = (request_rate / 1000) * req_max

Load estimate used in reward design:

load = (cpu_util + mem_util) / 2

Closest-state lookup during inference:

distance = |cpu1 - cpu2| + |mem1 - mem2| + |inst1 - inst2| + |req1 - req2|

Technical Flow When Input Is Given

  1. The user enters CPU, memory, running instances, and request rate in the React interface.
  2. The React frontend sends the request to the Node.js gateway endpoint.
  3. The Node.js gateway forwards that payload to the FastAPI backend.
  4. The FastAPI service normalizes the raw input into the same scale used by the trained samples.
  5. The backend compares the normalized state to the stored RL dataset and finds the closest matching sample.
  6. The action and reward associated with that learned sample are returned to the UI.
  7. The frontend presents the recommendation, reward, and debug information to the user.

Reward Logic

The reward function is designed to balance service quality and infrastructure cost. Its logic is based on system load, active instance count, and overload penalties.

  • If load > 0.75, scale_up gets a positive reward and other actions are penalized.
  • If load < 0.30, scale_down gets a positive reward and other actions are penalized.
  • If load is in the middle region, hold is favored.
  • A cost penalty is applied in proportion to the number of running instances.
  • An additional SLA-style penalty is applied if CPU or memory becomes extremely high.

Important Terms

  • State: the current condition of the system represented by CPU, memory, instances, and request rate after normalization.
  • Action: the scaling decision chosen by the policy: scale up, hold, or scale down.
  • Reward: a numerical score used to measure whether an action was beneficial for a state.
  • Policy: the rule used to decide which action should be selected in a given state.
  • Q-value: the estimated long-term usefulness of taking a particular action in a particular state.
  • Episode: one complete pass over the training dataset during RL training.
  • Alpha: learning rate controlling how strongly new information updates old Q-values.
  • Gamma: discount factor controlling how much future reward matters.
  • Epsilon: exploration parameter in epsilon-greedy learning; larger values promote experimentation, lower values promote exploitation.
  • Normalization: rescaling user input into the numeric space used by the trained data.
  • Inference: the deployed decision phase where the system recommends an action for a new input.
  • Closest sample matching: comparing a live normalized state to stored learned samples and using the nearest one as the recommendation source.

Why This Design Works

This approach keeps the deployed system simple and stable. Instead of retraining on every request, it uses knowledge derived from pre-trained workload behaviour. That makes the recommendation flow fast, repeatable, and easier to demonstrate.

It also clearly separates the stages of the project: offline RL experimentation and online inference.
